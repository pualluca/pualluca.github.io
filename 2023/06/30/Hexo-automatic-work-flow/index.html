Hexo automatic work flow

发表于 2023-06-30 更新于 2025-01-09 分类于 技术 阅读次数：

记录一下Hexo部署评论区的过程

Hexo使用感受

已经使用Hexo快要一年了，时间过的好快呀。

  • 最开始使用的时候，代码跑在本地，自己使用命令行，清除缓存，生成博客，再进行部署，是一个比较繁琐的流程，不能专注于写内容

  • 后来有一个契机，看到GitHub在自动集成部署方面做了很多的工作，actions可以很方便帮助用户进行代码的自动化部署，就开始尝试了一下

自动集成和部署

  • 借着GitHub pages的功能，可以方便展示自己的博客内容，不过有一个缺点，就是pages也是一个静态的网页，基于markdown的语法和文件，实际上有时候不那么容易去管理，而Hexo生成的东西是一些html的网页文件，如果在我的hexo项目中，每次写完博客，将生成的这些网页文件推送到GitHub pages库，就会非常麻烦，这怎么办呢。

  • 实际上hexo的配置文件config里面有一部分关于部署的，这个动作就可以自动将我们的hexo工程中生成的文件推送到我们目标的服务器上，那如何使这一步变成自动的呢，GitHub的action给我提供了一点思路，我每次提交的时候，让GitHub去触发这个action，在这个action里面，每次都生成博客内容，然后进行一键部署

  • 这一部分后面有空的话再更新吧，今天先不说了

Hexo集成评论区

  • 22年有一次早上写了Devotional笔记，想着可能有一些朋友想留言，从朋友那里了解到可以使用gitalk来做，后面就集成上了，不过一直没办法使用，一直显示了一个error，最近这几天自己的正事好像没啥心情去做，又不想再像以前一样浪费时间，想着要不我就把我的博客的评论区搞好吧，这样后面我再发一些内容，有的朋友想留下一些想法和意见的时候就可以随时留言

  • 然后就开始了为期3个下午的探索，下面就来记录一下

探索第一步

  • 1⃣️，将以前默认的图标修改成了最近新拍的一张比较满意的照片，把照片做成了一个小icon，引进到项目中来

  • 2⃣️，修改了一些配置，将未来评论区的内容的存储位置指定好，设置了一些token和密钥

  • 做完这些的时候，最明显的区别，就是评论区不再提示error了，而是如下图一样

探索第二步

  • 做完上面的内容之后，基本上就可以使用了，每次新发布一篇文章之后，我就以管理员的身份进入到我指定的存放评论的地方（也就是GitHub仓库的issue区），新建一个带有文章id和gitalk为label的issue，这样，评论就和文章关联起来了

  • 可是遵循极客原则的话，这就显得有点不够专业，并且以前发的每一篇博客都来这样搞一下，实在是挺大的工作量，为此，把这件事情让电脑去完成实在是非常有必要

  • 看了网上的一些人的分享，实现的基本原理就是通过站点（sitemap）的方式，对照着这位老哥的内容，基本上就可以实现自动生成了 hexo主题next中gitalk配置与评论初始化 - Coding World

小结

总结一下经验，大胆尝试，虽然踩了一些坑，有的代码的部分需要简单修改一下试一试总是可以的
